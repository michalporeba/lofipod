# Story 4.4: Migrate Supported Existing Data Without Data Loss

Status: review

## Story

As a developer upgrading a real app,  
I want supported local and remote todo data to migrate safely,  
so that version changes do not cause silent loss or ambiguity.

## Acceptance Criteria

1. Given a supported todo model change requires migration rather than simple read-through compatibility, when the developer upgrades the application to the new supported version, then the system provides a documented supported migration path for the affected local and remote data, and the migration remains within the bounded Phase 1 scope.
2. Given supported local todo data and supported canonical remote todo data exist before the upgrade, when migration is applied through the supported path, then the resulting data preserves the intended todo information without silent loss in normal supported scenarios, and the upgraded application continues to operate on the migrated result through the normal local-first workflow.
3. Given migration work changes either local representation, canonical remote representation, or both, when the developer inspects the upgraded state after migration, then the resulting data is consistent with the updated supported semantics, and the migration outcome is not left ambiguous or dependent on hidden one-off repair steps.
4. Given a migration cannot be fully completed within the documented supported model, when the system encounters that condition, then the incomplete or exceptional migration state is surfaced explicitly, and the behavior preserves trust by avoiding silent corruption or false claims of success.
5. Given the repository is being used to prove that evolution is safe rather than theoretical, when this story is complete, then developers can see a repeatable example of supported data migration without data loss in the bounded model, and that example provides the basis for later inspection and future-reuse stories.

## Tasks / Subtasks

- [x] Define and implement a bounded migration path from prior supported todo shape to the evolved shape for both local data and canonical-sync projection paths. (AC: 1, 2)
  - [x] Reuse existing reprojection/repair and sync orchestration paths instead of introducing ad hoc migration scripts.
  - [x] Keep migration semantics deterministic and idempotent across restarts and repeated reads.
- [x] Preserve canonical remote trust during migration. (AC: 1, 2, 3)
  - [x] Ensure migrated canonical Turtle remains compatible with current bounded semantics.
  - [x] Keep replication log and canonical updates consistent with existing push/pull ordering guarantees.
- [x] Surface unsupported/incomplete migration states explicitly. (AC: 4)
  - [x] Reuse existing classification/policy channels for unsupported shapes where possible.
  - [x] Avoid silent fallback that masks partial migration.
- [x] Add behavior-focused tests for successful and exceptional migration outcomes. (AC: 2, 3, 4, 5)
  - [x] Cover local migration, canonical migration, and replay/reconcile interaction.
  - [x] Cover repeatability and no-data-loss behavior for supported scenarios.
- [x] Update developer-facing docs to explain migration boundaries and inspectability expectations. (AC: 3, 5)
